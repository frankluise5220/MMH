# 开发文档与同步更新说明

本文档说明开发过程中哪些内容应该写进哪些文档，以及代码变化时需要同步更新哪些资料。目标是让 Web、iOS、Android、NAS 部署和后续维护都能沿着同一套产品语义推进。

信用卡分期规则与计算统一由 `src/lib/credit/installment.ts` 负责；数据结构在 `CreditCardInstallmentPlan` 与 `TxRecord.installment*` 字段中。修改分期费率、账期或冲抵逻辑时，同步检查账单汇总、交易 API、移动同步和删除/恢复联动。

## 文档分工

### `AGENTS.md`

用途：本地 agent 工作规则和长期产品/工程方向。

记录内容：

- 产品方向、数据层原则、API 原则、统一函数/模块方向。
- 反复确认过的长期决策，例如 Docker 更新方式、函数复用、局部刷新、Web/移动端分工。
- 不记录具体 host、账号、密码、SSH key、私有测试仓库地址等连接信息。

更新时机：

- 对话中形成了会长期影响后续开发的产品或工程决策。
- 发现 agent 经常重复犯同一类方向性错误。
- 文档分工、更新方式、API 规则、长期产品方向发生变化。

注意：

- `AGENTS.md` 是本地文件，不应作为公开安装说明。
- 私有地址、密码、测试密钥不能写入 `AGENTS.md`、公开文档或提交内容；需要临时使用时由当前会话或本机安全凭据提供。

### `docs/client-api.md`

用途：Web、iOS、Android 接入后端 API 的正式说明。

记录内容：

- 登录、服务器、账簿、用户、角色上下文。
- 通用返回格式、错误格式、日期/金额/分页/排序约定。
- 各模块 API：auth、overview、accounts、transactions、fund、settings、mobile sync。
- 每个接口的方法、路径、参数、返回结构和示例。

更新时机：

- 新增、删除或重命名 `/api/v1` 客户端接口。
- 修改接口参数、返回字段、错误格式、分页/排序规则。
- Web 和移动端都可能使用某个接口，或者移动端新增接入需求。
- 某个接口从 debug/internal 变为正式客户端契约。

### `docs/agent-api.md`

用途：Hermes、Codex、Claude 等外部 Agent 通过 HTTP 接入 MMH 后端的说明。

记录内容：

- 外部 Agent 可用的 Base URL、认证 header、通用返回格式。
- 可用的业务 API 和受控通用 DB API。
- 通用 DB API 的模型白名单、分页、账簿隔离和写入边界。
- 密钥、密码、token、邮箱账号、系统设置等敏感数据的访问禁区。

更新时机：

- 新增、删除或收紧外部 Agent 可调用接口。
- 修改 `/api/v1/db/models` 或 `/api/v1/db/data` 的认证、白名单、字段过滤、返回格式。
- 改变密钥存储、加密、脱敏、轮换或 Agent 权限策略。

### `docs/nas-install-manual.md`

用途：用户在 NAS 上安装和更新 MMH 的说明。

记录内容：

- Docker Compose 安装方式。
- GitHub/GHCR 正式源。
- 可选的本地测试源写法。
- 环境变量、端口、数据库、更新命令、常见问题。

更新时机：

- `docker-compose.yml`、`Dockerfile`、镜像名、端口、环境变量发生变化。
- 安装流程、更新流程、系统更新页面能力发生变化。
- 发布方式从本地测试源切换或同步到 GitHub/GHCR。

### `docs/security-hardening.md`

用途：远程登录、HTTPS、数据库端口、Prisma Studio、Cookie 和账簿隔离的安全边界说明。

记录内容：

- 哪些入口不能公开暴露。
- HTTPS、HSTS、Secure Cookie 的部署要求。
- 业务 API 必须使用当前 session 的账簿上下文进行过滤。
- 后续安全能力方向，例如登录限流、敏感字段加密、审计日志和会话管理。

更新时机：

- 登录/session Cookie、安全响应头、反向代理、数据库端口、Prisma Studio 暴露策略发生变化。
- 新增或收紧账簿隔离、审计、加密、限流等安全机制。

用途：Android 客户端打包、安装、发布说明。

更新时机：

用途：GitHub Release 对外发布说明模板。

## 开发时的同步更新规则

### 文档写法

面向用户或接入方的文档要少解说、多说明。用户打开文档时，首先需要知道怎么操作、填什么、点哪里、看到什么结果、出错后怎么办。

优先写：

- 操作步骤。
- 可复制的命令。
- 字段名和示例值。
- 成功后的预期结果。
- 常见错误和处理办法。
- 哪些内容需要用户选择。

少写：

- 长篇背景。
- 开发者内部推理。
- 与用户操作无关的技术解释。
- 重复说明“为什么这样设计”。

只有当原因会影响用户选择、避免误操作、或帮助判断故障时，才写简短解释。

### 本仓库工具使用约束

为减少编码、引号、正则和终端乱码带来的噪音，源码检查和修改默认遵循下面规则：

- 搜索优先使用 `rg` / `rg --files`。
- 读取源码文件时优先显式按 UTF-8 读取。
- 修改源码文件时优先使用 `apply_patch`。
- 不把 PowerShell 终端输出当作中文内容是否正确的依据，中文以文件实际内容为准。
- 避免用复杂 PowerShell 正则、嵌套引号、长链式表达式去处理源码；优先拆成多个小而稳定的读取/搜索步骤。
- 如果某个文件已经出现乱码或混合编码风险，先清理编码问题，再继续做逻辑修改。

### API 变化

同时检查：

- `src/app/api/v1/**/route.ts` 顶部 JSDoc。
- `docs/client-api.md`。
- Web 调用处。
- Android/iOS 调用处或移动端同步接口。

需要记录：

- 方法和路径。
- 请求参数、请求体。
- 返回字段。
- 错误格式。
- 是否需要登录、账簿、角色上下文。
- 是否影响分页、排序、日期、金额格式。

### 数据库和业务计算变化

同时检查：

- Prisma schema 和迁移。
- 统一服务模块，例如 fund、account、transaction 相关模块。
- Web 展示页和编辑页。
- 移动端 summary/sync 数据。
- `docs/check-investment-data.md` 或相关核对文档。

重点：

- 不要让 Web 和移动端各算一套。
- 不要让列表值、顶部汇总值、左侧概览值来自不同口径。
- 涉及余额时，不使用收入减支出替代账户余额。

### UI/交互变化

同时检查：

- 共享组件或页面组件。
- `docs/edit-window-checklist.md`。
- Web 和移动端是否需要不同布局但同一业务含义。

重点：

- 保存后是否局部刷新。
- 再记一笔是否保留上下文。
- SS 下拉是否支持搜索和新增。
- 数字表格是否保持列宽、排序、颜色规则一致。

### NAS 安装和更新变化

同时检查：

- `Dockerfile`
- `docker-compose.yml`
- `.env.example`
- `docs/nas-install-manual.md`
- `scripts/install-windows.bat`
- 系统设置里的系统更新页面

重点：

- 普通更新目标是拉取预构建镜像，不在 NAS 或普通用户本机上常规构建。
- 本地测试源和 GitHub/GHCR 正式源应尽量只差 URL。
- 不把本地私有 NAS 地址、密码、测试 API 写入公开文档。

### 发布前检查

发布前至少确认：

- 新接口是否补了 route JSDoc 和 `docs/client-api.md`。
- 安装或更新方式是否影响 `docs/nas-install-manual.md` 或 `scripts/install-windows.bat`。
- Android 相关变更是否影响 `docs/android-release.md`。
- 基金/投资计算变化是否更新 `docs/check-investment-data.md`。
- 新增/编辑窗口行为变化是否更新 `docs/edit-window-checklist.md`。
- `.env.example` 是否只包含公开安全的示例值。

### 导入调试日志

开发环境的账单/基金导入会把结构化诊断日志写入：

```text
.codex-logs/batch-import.debug.log
```

每次导入使用一个预览窗口可见的 `traceId`，可按该编号串联账户加载、文件解析、工作表合并、模式判断、校验阻断、批量修改、导入请求和数据库事务结果。

查看最近日志：

```powershell
Get-Content .codex-logs/batch-import.debug.log -Tail 100
```

查找一次导入：

```powershell
Select-String -Path .codex-logs/batch-import.debug.log -Pattern "traceId"
```

日志只能记录数量、模式、耗时、文件扩展名、文件大小、HTTP 状态和错误类型等诊断字段，不记录账单原文、文件名、账户名称、备注、密钥或密码。该文件仅在 `NODE_ENV=development` 时写入，并由现有 `*.log` Git 忽略规则排除。

## 信用卡独立账单与合并账单

- `Account.creditBillMode` 区分 `separate`（独立账单）和 `consolidated`（同机构合并账单）。
- 合并范围由“同一账簿 + 同一机构 + consolidated”确定，不额外维护账单组表。
- 同机构信用卡共用账单日、还款日和账单模式；修改其中一张卡时同步其他卡。
- 新增信用卡选择机构后，从该机构已有信用卡带出账单日、还款日、额度和账单模式。额度仅作为新卡默认值，不覆盖已有卡额度。
- 合并模式下，交易仍属于具体信用卡；账单汇总、明细、还款、周期和手工覆盖金额按组内所有卡计算。
- `CreditCardCycle` 和 `BillOverride` 使用组内按 ID 排序后的第一张信用卡作为稳定存储账户；停用卡仍保留在组内以保证历史账单不变化。接口和页面必须通过统一 helper 解析，不能直接假设当前点中的卡就是存储账户。
- 当前不建模共享授信、总授信或银行额度占用规则；额度仍是每个账户自己的字段。
