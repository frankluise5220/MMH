# Release 说明模板

发 GitHub Release 时，不要机械地每次都写长说明。

规则是：

- 小改动、小修补、内部整理，可以只发版本，不强制写详细说明。
- 只有大改动、用户可感知变化、安装/更新方式变化、兼容性变化，才建议认真写 Release 说明。
- 如果只是很小的修复，但你又想补一句，写一两行摘要即可。

## 适用场景

适合写 Release 说明的情况：

- 新增明显功能
- 页面结构或核心交互变化
- NAS / Docker / Android 安装更新方式变化
- 需要用户重新登录、重启、同步、迁移数据
- 有兼容性要求或已知问题需要提醒

不必单独认真写说明的情况：

- 小 UI 微调
- 文案修正
- 内部重构
- 小范围 bug 修补，且用户几乎无感

## 推荐标题

```text
v1.2.3
```

如果是 Android 单独发版，也可以用：

```text
android-v1.2.3
```

## 推荐正文模板

```md
## 本次更新

- 新增：
- 优化：
- 修复：

## 安装 / 更新说明

- Web / NAS：
- Android：

## 兼容性

- 需要的服务端版本：
- 是否需要重新登录：
- 是否需要重新同步缓存：

## 已知问题

- 暂无
```

## 最简短写法

如果这次改动不多，至少写成这样：

```md
## 本次更新

- 修复了：
- 优化了：

## 说明

- 这次更新是否需要重新登录 / 重启容器 / 重新同步
```

## GitHub 里怎么出说明

### 方式 1：手动写

创建 Release 时，直接把上面的模板贴进去改。

### 方式 2：用 GitHub 自动生成

仓库已经支持 `.github/release.yml`。

创建 Release 时点：

```text
Generate release notes
```

GitHub 会按标签把改动分组生成一版说明。然后再手工补一句更像人话的“本次更新”摘要，就比较完整了。

## 建议

- 面向用户的 Release 说明，优先写“这次能感知到什么变化”。
- 不要把内部重构、临时调试、无感知的小修补写得太重。
- 如果某次更新会影响 NAS 安装、Docker 更新、安卓兼容性，要单独写清楚。
