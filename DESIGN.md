# 显示层与编辑层规则

## 第一层：显示层

显示层变量 = f(数据库原始数据)

1. **Sidebar**：数据均从相关账户中直接获得，
   - 非投资类：从 `txRecord.groupBy` + `account.balance` 读取
   - 投资类：从 `fundHolding` 表 + `fundNavCache` 表读取（通过 `computeInvestBalances`）
2. **账户视图层**：各表格显示数据，也由各库中直接读出生成显示层变量
   - 持仓明细：统一从 `fundHolding` 表读取（通过 `computePositionDisplay`），禁止各页面自行从 entries 累加计算
   - 交易流水：从 `TxRecord` 读取（仅用于列表展示，不参与持仓计算）
3. 以上显示层变量在后续会因为编辑、新增、删除而被重新计算，从而导致数字的直接刷新

### 数据源单一性原则
- **投资类余额统一使用 `computeInvestBalances()`**，禁止各页面自行从 entries 累加
- **持仓明细统一使用 `computePositionDisplay()`**，数据源为 `fundHolding` 表
- **非投资类余额统一使用 `txRecord.groupBy` + `account.balance`**
- **`computeInvestBalances` 不再在读取时触发全量重算**，`recalcFundPositions` 仅在数据写入后调用

### 命名约定
- 显示层类型用 `Display` 后缀（如 `PositionDisplayRow`）
- 显示层变量在易混淆场景用 `display` 前缀
- 编辑弹窗 props 用 `current/initial` 前缀标注来源（如 `currentAmount`）
- 编辑弹窗内 useState 不需要前缀（作用域已清晰）

## 第二层：编辑层

编辑变量 = 临时 useState，生命周期 = 弹窗开→弹窗关

### 新增/编辑窗口使用单一组件
所有新增和编辑操作共用同一个组件，通过 `mode` 属性区分（`"create"` | `"edit"`）：
- **`InvestmentFormModal`**：基金投资的新增/编辑共用组件，create 模式显示"投资记账"，edit 模式显示"编辑基金记录"
- **`TransactionFormModal`**：普通消费/收入/转账的新增/编辑共用组件
- **`RegularInvestForm`**：定投计划的新增/编辑共用组件
- **禁止**创建分别的新增组件和编辑组件，必须用单一组件 + mode 参数
- **禁止**在 create 和 edit 之间复制字段逻辑，相同的字段使用同一套渲染代码
- edit 模式从 `entry` prop 初始化 useState；create 模式从 `defaults` prop 或空白初始化

1. **编辑弹窗打开时**：从数据层直接初始化 useState
2. **编辑弹窗关闭时**：`setOpen(false)` → 组件卸载/隐藏 → useState 自动销毁，无副作用
3. **编辑弹窗保存时**：
   - 调用 Server Action 或 API Route 写入 DB
   - 写入成功后调用 `revalidatePath`（Server Action）或确保 API Route 内有 `revalidatePath`
   - 客户端可选调用 `router.refresh()` 加速当前页面刷新
   - **禁止在保存前修改显示层变量**（不做乐观更新）

### 刷新触发链路

```
保存 → 写入DB → revalidatePath(相关路径) → Next.js缓存失效 → 下次请求时Server Component重新查库计算 → 显示层变量更新
```

## 编辑窗口通用做法
打开某一个编辑窗口时
1：从库里先读取数据
2：针对一些字段计算赋予默认值
3：用最新的数据更新窗口显示
4：保存的时候，将不管是计算出来的值，还是用户填的值，只要是与库里的值不一致的值，写回表里。然后刷新列表。

## 视图类型
目前共有下列种视图：
- detail: 借记卡、现金、电子钱包
- bill: 信用卡账单
- investfund: 开放式基金
- investmoney: 货币基金
- regularinvest: 定投计划

信用卡、开放式基金、货币基金应该分上下两块，
信用卡：上面是账户余额，下面是交易记录。
基金：上面是持仓信息，下面是交易记录。

### 持仓信息的计算与刷新
持仓信息的数值由 `recalcFundPositions` 从交易明细汇总后写入 `fundHolding` 表，显示层统一从 `fundHolding` 表读取（通过 `computePositionDisplay`）。
当基金交易明细有变化（新增/编辑/删除），先写入 TxRecord，再调用 `recalcFundPositions` 重新计算并刷新持仓库。

### 清仓基金切换
货币基金和开放式基金视图均支持切换到已清仓（units=0）的基金，查看其历史交易记录和编辑功能。清仓基金不在持仓列表中显示（持仓只显示有份额或有待确认金额的基金），但在持仓列表下方或通过下拉菜单可以切换到清仓基金的交易记录。

# 信用卡账单逻辑设计

## 1. 核心概念

### 账单周期
- 账单日（billingDay）：每月 N 号
- 还款日（repaymentDay）：每月 P 号
- 账单周期：上月账单日后一天~本月账单日
- 本期账单：最近一次已出账的月份 = currentStatementMonth 的上个月，不受隐藏按钮的影响，必须显示
- 未出账单：当前仍在进行中的账单周期所在的月份 = currentStatementMonth，不受隐藏按钮的影响，必须显示
- 0收支账单：即使继承了上期金额，只要收入和支出都为0的就叫0收支账单
- 示例：账单日 9 号 → 周期为 8月10日 ~ 9月9日，账单月在 9 月

### statementMonth
信用卡交易记录的 `statementMonth` 字段表示该交易所归属的**账单月**（即账周期结束的那个月）。

例如账单日 9 号：
- 2026-04-10 ~ 2026-05-09 的消费 → statementMonth = "2026-05"

## 2. 账户余额计算

### 非信用卡账户
balance = 该账户所有交易记录的金额总和（正向累加）

### 信用卡账户
balance = 本期账单 + 未出账单支出 - 未出账单收入（包括还款、退款、分期收入）**未还金额取负数显示**
balance是指最终的数字，只考虑本期账单和未出账单计算，不考虑以前的历史账单。

其中"账单金额" = 上期未还 + 该期支出 - 该期收入

## 3. 级联计算（effectiveBillByMonth）

对于每个账单月（按时间升序排列），计算该月的 effective bill：

```
effectiveBill[month] = override[month] ?? (effectiveBill[prevMonth] + rawBill[month])
```
刷新账单时：
从最早期的账单开始计算，
if最早期账单未经锁定
  计算该期账单金额（支出-收入）得到本期账单

往后循环
  if该期账单金额锁定
     跳下一期
  then
    计算账单金额=上期账单+该期账单金额（支出-收入）
    跳下一期
每当手动调整某期账单金额后
开始循环计算以后的账单，直至碰到锁定金额的账单停止

这样做是为了确保：即使某个月份没有记录，它的"上期未还"也会正确地流入下期。

### 示例
假设账单日 9 号，override 只锁定 2026-05 = 671.43：
```
2024-04: 上期账单=1577.53, 手动锁定=none → 支出=1000, 收入=0,   本期账单=2577.53
2024-05: 上期账单=2577.53,       手动锁定=none → 支出=0, 收入=0,   本期账单=2577.53  （无记录但有级联）
2026-05: 上期账单=2577.53,   手动锁定=671.43  → 支出=110, 收入=230,   本期账单=671.43  （账单金额取锁定值忽略所有本期数值）
2026-06: 上期账单=671.43,    手动锁定=none → 支出=100.11,  收入=0,   本期账单=771.54  （从671.43重新级联）
```

## 4. 级联值与余额的关系

左侧栏账户、概况、账户中心的金额 = balance

## 5. UI 显示

### 账单列表
- 本期账单行显示 `（本期账单）` 标签，不管本期帐单有没有收支，这一行都要显示。
- 未出账单行显示 `（未出账单）` 标签，不管未出帐单有没有收支，这一行都要显示。

### 标题金额
始终显示当前信用卡的 balance = 本期账单 + 未出账单支出 - 未出账单收入

### 筛选条件对余额的影响
余额计算与"隐藏0收支""隐藏已还清"等筛选条件无关，这些条件只影响显示的账单条数，不影响余额计算。

# 基金相关逻辑

## 核心架构：统一交易库

**基金明细已合并到 TxRecord**，不再有独立的 FundEntry 表。所有基金交易字段直接存储在 TxRecord 上：
- `fundCode`、`fundName` — 基金代码和名称
- `fundSubtype` — 交易子类型（buy/redeem/dividend_cash/dividend_reinvest/regular_invest/switch_in/switch_out/buy_failed）
- `fundUnits` — 成交份额（Decimal 20,6）
- `fundNav` — 成交净值（Decimal 20,6）
- `fundFee` — 手续费（Decimal 18,2）
- `fundArrivalAmount` — 到账金额（赎回时）
- `fundArrivalDate` — 到账日期（赎回时）
- `fundConfirmDate` — 确认日期
- `confirmDate` — 确认日期（别名）
- `realizedProfit` — 赎回收益
- `fundProductType` — 基金产品类型（fund/money/wealth/deposit）

基金交易的关联方式：通过 `accountId`（赎回/转出方）和 `toAccountId`（买入/转入方）关联资金账户，无 linkId/linkType 字段。

## 编辑窗口"获取"按钮逻辑

点击"获取"按钮时，系统执行以下操作：

1. **读取 T+N 参数**：
   - 优先从基金T+N信息库（FundConfirmDays）读取该基金该账户的确认天数
   - 若无配置，从账户默认确认天数（Account.defaultConfirmDays）读取
   - 若仍无，使用用户在界面填写的 T+N 值
   - 若用户填写了新值，保存到基金T+N信息库

2. **查找记录**：
   - 根据 entryId 查找 TxRecord（基金明细字段直接在 TxRecord 上）
   - 获取 TxRecord.date 作为申请日期

3. **计算确认日期**：
   - 确认日期 = 申请日期 + T+N 工作日
   - **跳过周末**：使用 addWorkdaysUtc 函数，只计算周一到周五

4. **获取净值**：
   - 调用天天基金 API，获取确认日期当天的净值
   - 若该日期无净值（非交易日/周末），返回错误提示

5. **计算份额**：
   - 份额 = (成交金额 - 手续费) / 净值（仅在净值已知时计算，净值未知时份额为0，用户也可手动填写）
   - 成交金额取绝对值

6. **更新界面字段**（不保存到数据库，但对净值库进行更新）：
   - 净值字段 ← API返回的净值
   - 份额字段 ← 计算的份额（净值已知时）
   - 确认日期字段 ← 计算的确认日期
   - 基金名称 ← API返回的名称

**注意**：获取按钮时只更新界面和净值库，持仓、交易明细不更新。用户需点击"保存"按钮才写入 TxRecord 并联动持仓基金的汇总更新。

### 界面字段布局

| 申请日期 || T+N | 确认天数，用于计算确认日期，可修改 | | 确认日期 |
| 基金代码 || 获取按钮 || 基金名称 |（可手动修改，或通过获取按钮自动填充 ）
| 手续费率(%) || 计算按钮 || 手续费 |（可手动填写，或点击"计算"按钮自动计算 ）

### 计算按钮逻辑
- 手续费率"计算"按钮：重新计算手续费（手续费 = 金额 × 手续费率）。当净值已知时，联动重新计算份额；净值未知时，仅计算手续费，份额保持为0。
- 净值"计算"按钮：当净值已知时，重新计算手续费和份额；净值未知时，无操作。

## 持仓基金库（FundHolding）计算规则

持仓基金库的份额/成本/均价由 `recalcFundPositions`（recalcPosition.ts）从 TxRecord 交易明细汇总后写入 `fundHolding` 表，显示层统一从 `fundHolding` 表读取。

### 写入逻辑
- 当持仓基金有成本或有份额，就在持仓列表中显示该基金
- 持仓成本 = 确认购买金额 + 待确认购买金额（不减手续费）
- 均价 = 已确认成本 / 已确认份额（不含待确认部分）
- 当交易明细有变化导致持仓基金库相关金额变化，刷新持仓显示同时刷新相关显示（如左侧基金账户）

### 基金名称来源
持仓名称采用三级优先级，切断明细→持仓→UI→明细的循环传播：
1. 权威来源：FundNavCache 中该基金代码的最新名称
2. 兜底1：TxRecord 中该基金代码的最近一条记录的 fundName
3. 兜底2：基金代码本身

### 显示规则
- 基金份额显示2位数，但数据库里记录6位小数（四舍五入）
- 持仓市值 = 已确认份额 × 最新净值 + 待确认购买金额
- 浮盈 = 市值 - 持仓成本（两者同步包含待确认部分）
- 持仓列表显示列：基金名称/份额/均价/净值/成本/市值/浮盈/浮盈率
- 市值列如果有待确认金额则显示"市值（-待确认）"，待确认金额以橙色(-XX.00)括号形式跟在数值后
- 历史收益是赎回时产生的收益

## 待确认成本
买入类型（buy/regular_invest/switch_in/dividend_reinvest）的交易，若尚未填写确认日期（fundConfirmDate 为 null）且净值为空（fundNav 为 null），视为"待确认"状态。
待确认金额 = 购买金额（不减手续费），在持仓市值旁边以 (-XXX.00) 橙色显示。
待确认的交易在交易明细中"确认日期"列显示"待确认"（橙色）；有净值但无确认日期的显示"—"。

## 成本摊薄方式（CostBasisMethod）
绑定在账户上，支持三种：
- moving_avg（移动加权平均）：每次买入后重新计算均价，赎回时按当前均价扣减成本
- fifo（先进先出）：按买入批次顺序依次消耗
- lifo（后进先出）：按买入批次逆序依次消耗

## 现金分红处理规则
dividend_cash（现金分红）在持仓计算中的处理：
- **减少持仓成本**但不减少份额（分红是从基金资产中派发的收益，不改变持有份额）
- dividend_cash 不属于买入类也不属于赎回类，recalcPosition.ts 需单独处理
- 关联操作：资金来源账户的 TxRecord 中增加一条投资类型、子类型为现金分红的记录，金额为分红金额

红利再投（dividend_reinvest）则增加份额（降低均价）。

## 赎回处理
赎回后，该基金的份额减少，持仓成本减少，持仓库的历史收益值变化。具体计算规则由 recalcPosition.ts 实现。
赎回收益（realizedProfit）由 recalcPosition.ts 按成本摊薄方式计算后写回 TxRecord。

## 基金库清单
1. **基金净值库**（FundNavCache）：代码，日期，基金净值。按日期排序，最新的净值在最后面。关联持仓基金用来显示最新市值。
2. **统一交易明细库**（TxRecord）：所有交易类型共用一张表，基金交易通过 `fundSubtype` 等基金专用字段区分。无独立的 FundEntry 表，无 linkId/linkType 双向关联。
3. **基金T+N信息库**（FundConfirmDays）：代码，账户ID，生效日期（初始为1970-01-01），确认天数。默认确认天数为0天。
4. **基金持仓库**（FundHolding）：由 recalcFundPositions 从 TxRecord 交易明细汇总后写入的持仓份额、持仓成本(含待确认金额）、均价等信息。均价 = 已确认成本 / 已确认份额。
5. **基金费率库**（FundFeeRate）：代码，账户ID，日期，费率类型（buy/redeem），基金费率。按日期排序，最新的费率在最后面。
6. **定投计划库**（RegularInvestPlan）：基金账户、资金账户、基金代码、金额、间隔、费率、确认天数等。

## 费率读取逻辑
费率库按基金+账户分组，每个基金每个日期只有一条费率。比如2025.1.1 0.1，2026.1.2 0.12，表示2025年1月1日到2026年1月2日的费率都是0.1，直到 2026.1.2，变成0.12。
到费率库中寻找该基金最新日期的记录：
1：取到，更新界面该字段
2：取不到，界面该字段置0
3：如果用户填写了费率，保存后，在库中查询确认日期以前是否有相同费率，如果有则不新增记录。如果该基金最近一条费率与当前保存的费率不相同，则新增记录（日期、费率）。

## 净值数据来源与缓存
在系统里设置设计添加获取基金参数API窗口及列表，支持增加多家API备用。以防止导入失败。
净值数据从 Eastmoney API 获取，使用以下字段：
- `DWJZ`：确认净值（已公布的单位净值）
- `jzrq`：确认净值日期
- `gsz`：实时估算净值（用于显示，不用于确认）

必须使用统一净值库模块（navCache.ts）操作净值数据，禁止直接 `prisma.fundNavCache.findUnique/upsert`。

## 基金明细编辑界面说明
界面中有三个按钮：
1："获取"，根据申请日期+N来计算确认日期，以确认日期去净值库中搜索相应的净值数据。如果没有，则以确认日期和基金代码去API获取净值，获取净值后，填回净值库。并覆盖编辑界面的净值字段、计算编辑层份额，并显示该值。
  如果手动修改确认日期，确认日期失去焦点后，则自动走获取净值的流程，跳过（根据申请日期+N来计算确认日期）的步骤
2："手续费率"后的"计算"按钮：重新计算手续费（手续费 = 金额 × 手续费率）。当净值已知时联动计算份额；净值未知时仅计算手续费。
3："净值"后的"计算"按钮：当净值已知时，重新计算手续费和份额；净值未知时无操作。

点击保存后，比较编辑层数值与库数据，将不同的数值重新写回 TxRecord、确认日期库。如果（资源来源、基金代码、申请日期、买入金额）发生了变化，需要在 TxRecord 对关联字段进行修改。手续费率变化时，在手续费库中查找该日记录，有命中则修改，无命中则追加。

## 资金来源账户关联
基金交易的资金来源通过 `toAccountId`（买入时资金流出方）或 `accountId`（赎回时资金流入方）关联。
编辑时如果资金来源字段从空变为有值，需要在资金来源账户中新增一条 TxRecord（类型=投资，子类型与基金子类型一致，金额为对应方向，对方账户为基金账户）。
如果资金来源字段只是修改（从A改为B），则在原资金来源账户的 TxRecord 中修改相关字段，同时在新资金来源账户新增一条 TxRecord。

## 基金持仓上面的获取净值按钮的说明
获取净值后有两个库要操作：
1：TxRecord 交易明细库，根据T+N策略寻找有未确认净值的数据，写入 fundNav（确认日净值）、fundUnits（份额）、fundConfirmDate（确认日期）。
2：调用 recalcFundPositions，从交易明细库重新计算并刷新基金持仓库的份额、均价、净值。
操作顺序很重要：先更新交易明细，再重新计算持仓。

注意：实时估算净值（gsz）和确认净值（dwjz）可能不同，应使用确认净值进行持仓计算。

## 投资账户市值计算
持仓基金的市值计算规则：
- 市值 = 已确认份额 × 最新净值 + 待确认购买金额
- 持仓成本 = 确认购买金额 + 待确认购买金额（不减手续费）
- 浮盈 = 市值 - 持仓成本
- 待确认状态：fundConfirmDate 为 null 的买入交易
- 均价 = 已确认成本 / 已确认份额（不含待确认部分）
持仓显示规则：
- 市值金额后面跟随未确认金额(xxx.xx)
